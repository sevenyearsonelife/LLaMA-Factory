# DPO Training Metrics & Batch Operations

## Q: 在DPO训练过程中，我应该关注哪些指标？该如何配置这些指标呢？

在 DPO（Direct Preference Optimization）训练过程中，你无法“直接”配置这些指标（Metrics），因为它们是根据当前策略模型（Policy Model）和参考模型（Reference Model）的输出客观计算得出的监控值。不过，你需要**密切监控特定的核心指标**，并通过**调整超参数**来干预和优化这些指标的走向。

基于 LLaMA-Factory 的源码和工业界最佳实践，以下是 DPO 训练时的核心监控重点与配置优化方案：

### 一、 核心监控指标 (Metrics to Monitor)

在训练日志（如终端输出、WandB 或 TensorBoard）中，你应重点关注以下 `rewards` 和 `logps` 相关指标：

1. **`rewards/accuracies` (奖励准确率 - 核心指标)**
   - **含义**：当前模型对 chosen（偏好回答）的隐式奖励打分高于 rejected（拒绝回答）的频率。
   - **预期表现**：从 0.5 左右稳步上升，理想状态收敛在 `0.75 - 0.95` 之间。如果迅速飙升逼近 1.0，说明模型可能过拟合了训练集（例如利用了某种文本长度的捷径）。
2. **`rewards/margins` (奖励边距)**
   - **含义**：`chosen_rewards` 与 `rejected_rewards` 的平均差值。
   - **预期表现**：应从 0 附近逐渐平稳上升到一个正数（如 `1.0 ~ 3.0`）。如果上升过快过大，意味着模型为了拉开差距正在极度打压坏回答，可能导致生成的文本严重扭曲不自然。
3. **`rewards/chosen` 与 `rewards/rejected`**
   - **含义**：模型给两者分配的绝对隐式奖励值。
   - **预期表现**：通常 `chosen` 稳步上升或持平，`rejected` 明显下降。如果两者同时断崖式下跌，代表模型正在极速偏离 Reference Model，出现了明显的对齐崩溃（KL 崩坏）。
4. **`logps/chosen` 与 `logps/rejected` (对数概率)**
   - **含义**：当前模型生成这两个回复的对数概率。
   - **预期表现**：在 DPO 中这两者通常都会下降（因为模型分布在改变），但 `rejected` 下降的速度必须比 `chosen` 快得多。如果 `logps/chosen` 暴跌，意味着模型“不会说话了”，丢失了 SFT 阶段学到的语言连贯性。

### 二、 如何通过配置干预指标走向 (Hyperparameter Configuration)

当你发现上述监控指标走势不健康（如 logps 崩盘、accuracy 迟迟不涨）时，需要修改你的 YAML 配置文件中的参数进行干预：

#### 1. 调整 KL 散度惩罚：`pref_beta` (默认: `0.1`)
*这是 DPO 中最关键的参数，控制策略模型偏离参考模型时的惩罚力度。*
- **当 `logps/chosen` 断崖式下跌，生成文本乱码**：说明模型为了拉大 margin，不顾一切地改变了词汇分布。你需要**调大 `pref_beta`** (例如调到 `0.2` ~ `0.5`)，强制模型更紧贴参考模型。
- **当 `rewards/accuracies` 几乎不涨，`margin` 极小**：说明模型优化过于保守。可以尝试**调小 `pref_beta`** (例如调至 `0.05` 甚至 `0.01`)，允许模型更大胆地偏向 chosen 数据。

#### 2. 拯救语言生成连贯性：`pref_ftx` (默认: `0.0`)
- **场景**：如果虽然 accuracy 指标很好，但推理时模型经常跑题或出现语法错误（灾难性遗忘）。
- **配置**：将其设为 `0.1` ~ `0.5` 之间。这会在 DPO 损失之外混合一个标准的 SFT 交叉熵损失（SFT loss term），强迫模型在学习偏好的同时不忘掉基本的语言建模能力。

#### 3. 缓解偏好数据的噪音过拟合：`dpo_label_smoothing` (默认: `0.0`)
- **场景**：发现 `loss` 震荡严重，且已知你的偏好数据集（Chosen/Rejected）内部有标注冲突或脏数据。
- **配置**：可以设为 `0.1` 或 `0.2`（取值限制在 0~0.5 之间）。它使用了 cDPO（Conservative DPO）思想，通过标签平滑防止模型对存在瑕疵的偏好标注过度自信。

#### 4. 更换底层损失算法：`pref_loss` (默认: `sigmoid`)
改变底层的损失函数会直接改变这些指标的计算与优化目标逻辑。
- **`sigmoid`**：标准 DPO。
- **`simpo`**：使用 SimPO 算法，无需加载 Reference Model。可以通过设置 `simpo_gamma` (默认 `0.5`) 参数来硬性规定目标奖励边界。如果标准 DPO 下 margin 总是拉不开，可以尝试切换到此模式。
- **`orpo`**：使用 ORPO 算法，无需 Reference Model。开启后，系统指标将不再关注 reference logps，而是输出 `sft_loss` 和 `odds_ratio_loss` 供你监控。

## Q: 在tensorboard上，给出上述指标的ascii曲线示意图

在 TensorBoard 或 WandB 上，一个**健康且收敛良好**的 DPO 训练过程，其核心指标曲线通常会呈现如下的走势。

这里为你绘制了这 4 个核心监控面板的 ASCII 示意图：

```text
=========================================================
1. rewards/accuracies (偏好准确率)
=========================================================
1.00 | 
     |                       ,----------* (理想收敛区 0.75~0.95)
     |                    ,-'
0.75 |                 ,-'
     |              ,-'
0.50 | *-----------'
     |
0.00 +------------------------------------- Steps
     ^
    训练初期模型分不清好坏，准确率在 0.5 左右随机波动。
    随后稳步爬升，并在 0.8 左右趋于平缓（过高说明过拟合了数据特征）。

=========================================================
2. rewards/margins (奖励边距)
=========================================================
3.00 |                       ,----------* (拉开了稳健的奖励差距)
     |                    ,-'
     |                 ,-'
1.50 |              ,-'
     |           ,-'
0.00 | *--------'
     |
     +------------------------------------- Steps
     ^ 
    刚开始 policy 和 reference 模型一样，margin 为 0。
    随着训练进行，模型逐渐拉开 chosen 和 rejected 的打分差距。

=========================================================
3. rewards/chosen 与 rewards/rejected (绝对奖励值)
=========================================================
     |
 2.0 |                       ,----------* rewards/chosen (稳步上升)
     |        chosen      ,-'
 0.0 | *-----------------'
     |  `-.   rejected
-2.0 |     `-.
     |        `-.
-4.0 |           `----------------------* rewards/rejected (显著下降)
     |
     +------------------------------------- Steps
     ^
    这是分解后的奖励图：模型会轻微“奖赏”好回答（蓝线向上），
    但主要是通过强烈“打压”坏回答（红线向下）来拉开两者的差距。
    只要两者没有同时雪崩式暴跌，就是健康的。

=========================================================
4. logps/chosen 与 logps/rejected (对数概率)
=========================================================
     |
     | *--.   logps/chosen
-100 |     `--.-------------------------* (缓慢轻微下降)
     |         \
     |          \
     |           \    logps/rejected
-150 |            `--.
     |                `-----------------* (快速大幅下降)
     +------------------------------------- Steps
     ^
    在 DPO 中，由于模型偏离了原始参考模型，两者的生成概率都会下降。
    关键在于：坏回答 (rejected) 的生成概率必须暴跌，
    而好回答 (chosen) 的概率只允许轻微下降。
    (注：如果 logps/chosen 也跟着暴跌，说明模型丢失了说人话的能力，需要调大 pref_beta)
```

### 诊断小贴士：
如果你在实际 TensorBoard 面板上看到的曲线与上述 ASCII 图严重不符：
- **Accuracies 呈一条直线死死卡在 0.5**：学习率 (learning rate) 太低，或者 `pref_beta` 太大导致模型完全不敢动。
- **Accuracies 开局直冲 0.99**：大概率是数据泄露（例如 chosen 永远比 rejected 长，模型只学到了字数越长越好）。
- **Chosen Logps 断崖式直线暴跌**：模型正在经历“对齐税（Alignment Tax）灾难”，它为了迎合偏好完全打破了语言分布，这时候你需要立刻调大 `pref_beta`，或者开启 `pref_ftx` 引入 SFT 损失保底。

## Q: 以rewards/chosen指标为例，每一个step使用多少数据进行计算？

在 TensorBoard 上看到的 `rewards/chosen` 指标数据点，并不是由固定的一条或几条数据算出来的。从底层的张量计算，到最终在面板上画出一个点，经历了 **三次维度的求平均（Aggregation）**。

在 LLaMA-Factory 的源码逻辑中（基于 `trl` 的 `DPOTrainer` 与 `CustomDPOTrainer`），每一个点背后使用的数据量是由你的**训练超参数共同决定的**。

我们可以顺着代码逻辑，把这个过程拆解开来：

### 1. 第一次平均：微批次级（Micro-Batch）
前向传播（Forward Pass）发生在每一次梯度计算时。在 `src/llamafactory/train/dpo/trainer.py` 的 `get_batch_loss_metrics` 函数中：
```python
chosen_rewards = self.beta * (chosen_logps.to(device) - ref_chosen_logps.to(device)).detach()
# ...
metrics[f"{prefix}rewards/chosen"] = chosen_rewards.mean().item()
```
这里 `chosen_rewards` 是一个一维张量（Tensor），长度等于**单卡微批次大小 (`per_device_train_batch_size`)**。
这意味着，在最低维度上，它是对当前这 1 张 GPU 上塞进去的 `N` 对 (Chosen, Rejected) 样本算出的 reward 进行了一次 `mean()` 取平均，并将这个标量缓存进内存。

### 2. 第二次平均：时间与步数累加（Steps）
模型并不是做一次前向传播就在 TensorBoard 上打一个点。
HuggingFace Trainer 会通过 `logging_steps` 参数控制打点频率（默认比如每 10 步记录一次）。
此外，每次权重更新（Global Step）可能包含多次前向传播（由 `gradient_accumulation_steps` 决定）。

这段时间内产生的所有标量都会被暂存在 `_stored_metrics` 列表中。当触发打日志时，`CustomDPOTrainer.log()` 会把这段时间攒下来的指标再求一次平均：
```python
metric_list.append(torch.tensor(metrics, dtype=torch.float).to(self.accelerator.device).mean().item())
```
这包含了 `gradient_accumulation_steps * logging_steps` 次前向传播的数据。

### 3. 第三次平均：跨设备同步（Distributed/GPUs）
如果使用了多卡（DDP/DeepSpeed），不同显卡看到的数据是不一样的。在 `log()` 函数的最后，有一步 Reduce 操作：
```python
metric_list = self.accelerator.reduce(metric_list, "mean").tolist()
```
系统会将所有 GPU 节点计算出的平均值收集起来，再做一次全局平均，最终推送到 TensorBoard。

---

### 💡 结论：一个点背后到底有多少数据？

综合上面的代码流转，TensorBoard 上**每产生一个新的数据点**，其实际反映的 **Chosen 样本总数** 公式为：

**计算数据量 = `per_device_train_batch_size` × `gradient_accumulation_steps` × `GPU_Nums` × `logging_steps`**

#### 举个具体例子：
假设你的 YAML 配置如下：
- `per_device_train_batch_size: 2` （每张卡每次塞入 2 个对话对）
- `gradient_accumulation_steps: 4` （累积 4 次才更新一次模型，Global Batch Size 单卡等效为 8）
- 使用了 **8 张 GPU** 训练
- `logging_steps: 10` （TensorBoard 上的 Step 轴，每隔 10 步新增一个点）

那么，TensorBoard 上位于 `step = 10` 的那个点上的 `rewards/chosen` 值，代表了：
`2 × 4 × 8 × 10 = 640` **条 Chosen 样本的整体平均 Reward**。

**诊断意义**：
了解这一点很重要，因为当你的 `logging_steps` 设得比较大时，TensorBoard 的曲线会非常平滑，但可能会掩盖掉某一个极度糟糕的数据对（Outlier）引发的瞬间 Loss 尖峰。如果你需要非常精细的除错，可以临时调小 `logging_steps`。

## Q: 我不太清楚train的batch和eval的batch是什么关系？请给出一个train loop代码示例进行讲解

在深度学习中，**Train Batch（训练批次）** 和 **Eval Batch（验证/评估批次）** 在数据结构上是完全一样的（都是包含 input_ids, labels 等张量的字典），但它们在整个训练生命周期中的**作用、来源和计算逻辑有着根本的区别**。

简单来说：
- **Train Batch 负责“学习”**：来自训练集。模型需要对它计算损失（Loss）、反向传播求梯度（Gradients），并更新自己的神经网络权重（Weights）。
- **Eval Batch 负责“考试”**：来自验证集。模型只对它做前向传播（Forward），计算一下当前的指标（比如准确率、验证集 Loss），用来观察模型有没有“死记硬背”（过拟合）。**绝对不能用它来更新权重。**

下面我用一段极简的 PyTorch 风格的 Training Loop 伪代码来为你演示它们的关系与交替过程：

```python
import torch

# 假设我们有两个数据加载器：一个装训练题，一个装期中考试题
train_dataloader = get_dataloader(train_dataset, batch_size=4)
eval_dataloader = get_dataloader(eval_dataset, batch_size=4)

# 超参数
num_epochs = 3
eval_steps = 100  # 每训练100步，就停下来考一次试
global_step = 0

for epoch in range(num_epochs):
    # ==========================================
    # 1. 训练阶段 (Training Phase)
    # ==========================================
    model.train()  # 开启训练模式 (启用 Dropout 等)
    
    for train_batch in train_dataloader:
        # 将数据放到显卡上
        train_batch = {k: v.to('cuda') for k, v in train_batch.items()}
        
        # 前向传播 (计算预测结果)
        # 注意：这里 PyTorch 默认会记录计算图，准备求梯度
        outputs = model(**train_batch)
        train_loss = outputs.loss
        
        # 反向传播 (算误差梯度)
        train_loss.backward()
        
        # 优化器更新权重 (模型真正变聪明的地方)
        optimizer.step()
        optimizer.zero_grad()  # 清空梯度，为下一步做准备
        
        global_step += 1
        print(f"Step {global_step} | Train Loss: {train_loss.item()}")
        
        # ==========================================
        # 2. 评估阶段 (Evaluation Phase) - 周期性触发
        # ==========================================
        if global_step % eval_steps == 0:
            print("--- 开始周期性验证 (Evaluation) ---")
            
            model.eval()  # 切换到评估模式 (关闭 Dropout 等，保证稳定)
            total_eval_loss = 0.0
            
            # 使用 torch.no_grad() 告诉 PyTorch：
            # "现在是考试，不要记录计算图，不要算梯度，省点显存！"
            with torch.no_grad():
                for eval_batch in eval_dataloader:
                    # 验证集的 batch 进入模型
                    eval_batch = {k: v.to('cuda') for k, v in eval_batch.items()}
                    
                    # 仅做前向传播
                    eval_outputs = model(**eval_batch)
                    total_eval_loss += eval_outputs.loss.item()
            
            avg_eval_loss = total_eval_loss / len(eval_dataloader)
            print(f"Step {global_step} | Eval Loss: {avg_eval_loss}")
            
            # 记录到 TensorBoard (这就是你看到的 eval_loss 曲线)
            tensorboard.log({"eval_loss": avg_eval_loss}, step=global_step)
            
            # 考完试了，切回训练模式，继续回去做题
            model.train()
```

### 总结它们的关系：

1. **并行而不相交**：在 `trainer.py` 中，虽然 `get_batch_loss_metrics` 这个函数既能算 Train 的奖励，也能算 Eval 的奖励（通过传入 `train_eval="eval"` 参数区分），但底层输入的数据流是完全分开的。
2. **频率不同**：Train Batch 是每个 Step 都在流水线里跑的；而 Eval Batch 是一次性抓取（或抽取一部分）验证集，在特定的节点（比如每 500 步）集中跑一遍。
3. **TensorBoard 表现**：
   - `train/loss` 或 `train/rewards/chosen` 是**高频**波动的曲线，代表日常作业成绩。
   - `eval/loss` 或 `eval/rewards/chosen` 是**低频**（点比较稀疏）的曲线，代表阶段性统考成绩。如果你发现 Train 成绩越来越好，但 Eval 成绩变差，说明模型“做题家”附体，过拟合了。
